# **Документація юз-кейсів**

## **Use Cases**

### 1. Реєстрація користувача

- Актор: Гість

- Передумови:

- Користувач не має акаунта в системі.

**Основний сценарій:**

- Користувач відкриває сторінку реєстрації.

- Вводить ім’я, прізвище, email, телефон, дату народження та пароль.

- Система перевіряє валідність даних.

- Система створює нового користувача у базі даних.

- Користувач може виконати вхід у систему.

**Альтернативні сценарії / помилки:**

- Email вже використовується.

- Пароль коротший за 8 символів.

- Пароль не містить цифр.

### 2. Вхід у систему

- Актор: Гість

- Передумови:

- Користувач зареєстрований у системі.

**Основний сценарій:**

- Користувач відкриває сторінку входу.

- Вводить email і пароль.

- Система перевіряє дані.

- Якщо дані правильні — користувач входить у систему.

- Система створює cookie з userId.

**Альтернативні сценарії / помилки:**

- Неправильний email.

- Неправильний пароль.

### 3. Перегляд каталогу товарів

- Актор: Гість, Клієнт

- Передумови:

- У системі є активні товари.

**Основний сценарій:**

- Користувач відкриває головну сторінку.

- Система показує список товарів.

- Користувач може обрати категорію.

- Система відображає товари вибраної категорії.

- Користувач може переходити між сторінками каталогу.

**Альтернативні сценарії / помилки:**

- Вибрана категорія не містить товарів.

### 4. Додавання товару до кошика

- Актор: Клієнт

- Передумови:

- Користувач авторизований.

- Товар існує.

**Основний сценарій:**

- Користувач обирає товар.

- Натискає кнопку "Додати до кошика".

- Система перевіряє чи існує кошик користувача.

- Якщо кошика немає — система створює його.

- Товар додається до кошика.

- Якщо товар уже є у кошику — збільшується кількість.

**Альтернативні сценарії / помилки:**

- Користувач не авторизований ---> перенаправлення на сторінку входу.

### 5. Перегляд кошика

- Актор: Клієнт

- Передумови:

- Користувач авторизований.

**Основний сценарій:*

- Користувач відкриває сторінку кошика.

- Система отримує товари кошика з бази даних.

- Система показує список товарів і їх кількість.

**Альтернативні сценарії / помилки:**

- Кошик порожній.

### 6. Зміна кількості товару у кошику

- Актор: Клієнт

- Передумови:

- Користувач авторизований.

- Товар уже знаходиться у кошику.

**Основний сценарій:**

- Користувач змінює кількість товару.

- Система оновлює кількість у базі даних.

- Кошик оновлюється.

**Альтернативні сценарії / помилки:**

- Кількість = 0 ---> товар видаляється з кошика.

### 7. Видалення товару з кошика

- Актор: Клієнт

- Передумови:

- Товар знаходиться у кошику.

**Основний сценарій:**

- Користувач натискає кнопку видалення товару.

- Система знаходить товар у кошику.

- Система видаляє його з бази даних.

**Альтернативні сценарії / помилки:**

- Товар уже відсутній у кошику.

### 8. Створення замовлення

- Актор: Клієнт

- Передумови:

- Користувач авторизований.

- У кошику є товари.

**Основний сценарій:**

- Користувач відкриває сторінку оформлення замовлення.

- Вводить дані отримувача.

- Обирає спосіб доставки.

- Система створює нове замовлення.

- Система створює записи OrderItems.

- Кількість товарів на складі зменшується.

- Кошик очищується.

**Альтернативні сценарії / помилки:**

- Кошик порожній ---> замовлення не створюється.

### 9. Перегляд історії замовлень

- Актор: Клієнт

- Передумови:

- Користувач авторизований.

**Основний сценарій:**

- Користувач відкриває сторінку своїх замовлень.

- Система отримує список замовлень користувача.

- Система відображає замовлення у хронологічному порядку.

**Альтернативні сценарії / помилки:**

- Замовлень ще немає.

### 10. Керування товарами (адміністратор)

- Актор: Адміністратор

- Передумови:

- Адміністратор авторизований.

**Основний сценарій:**

- Адміністратор відкриває адмін-панель.

- Створює новий товар або редагує існуючий.

- Система зберігає зміни у базі даних.

**Альтернативні сценарії / помилки:**

- Некоректні дані товару.

### 11. Керування замовленнями (адміністратор)

- Актор: Адміністратор

- Передумови:

- Адміністратор авторизований.

**Основний сценарій:**

- Адміністратор відкриває список замовлень.

- Переглядає деталі замовлення.

- Змінює статус замовлення.

- Система зберігає новий статус.

**Альтернативні сценарії / помилки:**

- Замовлення не знайдено.
